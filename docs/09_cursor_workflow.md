# 09. CURSOR WORKFLOW

## 전제

이 프로젝트는 Cursor AI로만 진행할 예정이므로,
중요한 것은 “좋은 기획”보다 **좋은 작업 분해와 전달 방식**이다.

Cursor는 한 번에 많은 걸 시키면 흐려진다.
따라서 아래 원칙을 지켜야 한다.

---

## Cursor 운영 핵심 원칙

### 1. 한 번에 하나의 목적만 시킨다
좋은 예:
- 2048 이동 엔진만 구현
- 로비 자원바만 구현
- 저장 시스템만 구현

나쁜 예:
- 게임 전체를 예쁘게 완성해줘

---

## 2. 항상 파일 범위를 지정한다
좋은 예:
- `features/puzzle/engine/*`
- `components/puzzle/GameBoard.tsx`
- `stores/usePuzzleStore.ts`

나쁜 예:
- 필요한 파일 다 알아서 수정해

---

## 3. 완료 기준을 준다
좋은 예:
- 방향키와 스와이프 둘 다 동작
- 같은 타일은 한 턴에 한 번만 합쳐짐
- 게임오버 판정까지 구현

---

## 4. 시각 톤을 구체적으로 준다
좋은 예:
- 몰랑한 카드형
- 따뜻한 크림/브라운
- 버튼 눌림 반응
- 180~250ms 부드러운 트랜지션

---

## 5. 코드를 고치기 전에 분석부터 시킬 때와 바로 작업시킬 때를 구분한다

### Ask / Chat 모드에 적합
- 구조 분석
- 리팩터링 방향
- 작업 분해
- 버그 원인 추정

### Agent / Edit 모드에 적합
- 실제 파일 수정
- 기능 구현
- 스타일 정리
- 리팩터링 적용

---

## 대화창 분리 추천

프로젝트를 아래 단위로 대화창을 분리하는 것이 좋다.

### 대화창 A. 코어 엔진
- 2048 로직
- 상태 관리
- 저장 로직

### 대화창 B. UI 컴포넌트
- 버튼
- 카드
- 모달
- 자원바
- 탭

### 대화창 C. 로비/경영
- 로스터
- 생산
- 진열
- 고객

### 대화창 D. 모션/폴리시
- 미세 애니메이션
- 전환
- 사운드
- 반응감

### 대화창 E. 문서/기획
- docs 업데이트
- prompts 업데이트
- TODO 정리

이렇게 나누면 맥락 오염이 줄어든다.

---

## 추천 작업 흐름

### Step 1. 먼저 분석시킨다
예:
- 현재 구조를 보고 2048 엔진 구현에 필요한 파일만 추려줘
- 지금 상태에서 가장 안전한 구현 순서를 제안해줘

### Step 2. 작업 단위를 자른다
예:
- 이번 작업은 2048 이동/합체 엔진만
- UI는 건드리지 말고 순수 함수로 구현

### Step 3. 수정 범위를 명시한다
예:
- features/puzzle/engine/*
- tests/puzzle/*
- 기존 디자인 시스템 파일은 수정하지 마라

### Step 4. 검수 포인트를 준다
예:
- 이동 규칙이 표준 2048과 같아야 함
- 한 번 이동에 이중 합체 금지
- 테스트 코드 포함

### Step 5. 완료 후 요약을 요구한다
예:
- 변경 파일 목록
- 구현 요약
- 남은 리스크
- 다음 작업 추천

---

## 커서에게 항상 요구하면 좋은 출력 형식

아래 형식을 습관적으로 요구하면 좋다.

### 출력 형식
1. 작업 요약
2. 변경 파일 목록
3. 구현 내용
4. 검증 방법
5. 남은 이슈
6. 다음 추천 작업

## 좋은 전달문 예시

### 예시 1. 엔진 구현
```md
이번 작업은 2048 이동/합체 엔진만 구현해.
UI는 최소 수정만 하고, 핵심은 순수 함수 엔진 분리야.

수정 범위:
- features/puzzle/engine/*
- features/puzzle/model/*
- stores/usePuzzleStore.ts
- tests/puzzle/*

요구사항:
- 4x4 보드
- 상하좌우 이동
- 같은 값 타일 합체
- 한 턴에 한 번만 합체
- 새 타일 스폰
- 게임오버 판정
- 테스트 포함

스타일:
- 코드 깔끔하게 분리
- 타입 명확하게
- UI 종속 최소화

완료 후 아래 형식으로 보고:
1. 작업 요약
2. 변경 파일 목록
3. 구현 내용
4. 테스트 방법
5. 남은 리스크
```

### 예시 2. UI 작업
```md
이번 작업은 퍼즐 보드 UI와 타일 애니메이션만 개선해.
엔진 규칙은 건드리지 말고, 시각 반응만 강화해.

수정 범위:
- components/puzzle/*
- app/puzzle/page.tsx
- lib/easing.ts

목표:
- 타일 생성/이동/합체 애니메이션 강화
- 몰랑한 반응
- 고급스럽고 저자극
- 모바일 기준 부드럽게

금지:
- 전체 구조 갈아엎기
- 엔진 로직 변경
```

---

## 가장 중요한 운영 팁

### 1. 처음부터 완성형 요구 금지
항상 작은 단위로.

### 2. “왜”를 먼저 설명
단순 구현보다 의도를 줘야 품질이 오른다.

### 3. 시각 톤을 계속 반복
이 프로젝트의 강점은 감성/반응감이므로 계속 상기시켜야 한다.

### 4. 수정 금지 범위를 적는다
안 그러면 필요 없는 파일까지 건드린다.

### 5. 작업 후 테스트 방법을 반드시 받는다
초보자일수록 이 부분이 중요하다.

---

## 추천 세션 마무리 방식

매번 마지막에 아래를 받는다.

- 현재 완료 상태
- 다음 해야 할 3개 작업
- 지금 바로 테스트할 체크리스트
- 회귀 위험 포인트

---

## 결론

Cursor를 잘 쓰는 핵심은
**“똑똑한 질문”이 아니라 “좋게 쪼갠 작업”**이다.

이 프로젝트에서는 특히 아래가 중요하다.

- 범위 제한
- 파일 지정
- 완료 기준 명시
- 감성 키워드 반복
- 문서 업데이트 동반
